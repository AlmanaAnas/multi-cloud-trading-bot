How it works

From Binance to BigQuery
in five steps

Everything runs automatically, every minute, with no servers to manage.

01
EventBridge fires a timer
AWS EventBridge works like a cron job. Every 60 seconds it wakes up the Lambda function. Nothing runs between triggers — no servers, no cost.
AWS EventBridge
rate(1 minute)
02
Lambda pulls live BTC prices from Binance
The Lambda function calls Binance's free public API — no API key needed. It fetches the last 100 one-minute candles for BTC/USDT: Open, High, Low, Close, and Volume for each minute. From those 100 candles it computes RSI (momentum), ATR (volatility), and MA20/MA50 (trend direction).
AWS Lambda · Python 3.11
api.binance.com
03
Data crosses from AWS to GCP — without any keys
Lambda publishes the payload to GCP Pub/Sub. The authentication uses Workload Identity Federation — Lambda exchanges its AWS identity for a short-lived GCP token that expires in 1 hour. No passwords. No API keys. No JSON files stored anywhere.
Workload Identity Federation
WIF · STS → GCP token
04
Cloud Function runs the ML model
GCP Pub/Sub instantly pushes the message to the Cloud Function. It loads the pre-trained model from GCS, runs prediction on the RSI/ATR/MA features, and calculates: direction (LONG or SHORT), confidence score (0–100%), and stop-loss price (entry − ATR × 1.5).
GCP Cloud Function · Python
model.predict(features)
05
Signal archived — alert fires if confidence > 80%
Every signal gets written to BigQuery for historical analysis and future model retraining. If the model confidence is above 80%, a Telegram alert is sent with the direction, entry price, and stop-loss level.
BigQuery · Telegram Bot

Binance provides public market data endpoints that anyone can call without registering or providing an API key. The Lambda function calls GET /api/v3/klines?symbol=BTCUSDT&interval=1m&limit=100 and receives back an array of 100 candles. Each candle contains the open, high, low, close prices and volume for that one-minute window. The Lambda then computes RSI, ATR, and moving averages from these 100 data points using pure Python — no external libraries needed.

Security

Zero credentials stored anywhere

Every authentication boundary in the system uses short-lived tokens. No passwords. No keys. No JSON files.

🔑
GitHub → AWS
GitHub Actions gets a short-lived OIDC token from GitHub, then uses it to assume an IAM role in AWS. The role is locked to this exact repository — nothing else can use it.
✓ No AWS_ACCESS_KEY_ID stored
🌉
GitHub → GCP
Workload Identity Federation maps GitHub's OIDC token directly to a GCP service account. The attribute condition is locked to this repository path.
✓ No service account key file
⚡
Lambda → GCP
At runtime Lambda exchanges its AWS identity for a short-lived GCP token using Workload Identity Federation. The token expires in 1 hour. Nothing is stored.
✓ Keyless cross-cloud auth
🛡️
Least privilege IAM
Every role has exactly the permissions it needs. Lambda can only publish to Pub/Sub. The CI role can only manage the specific services this project uses.
✓ No broad admin roles at runtime
🗄️
Encrypted state
Terraform state is encrypted at rest in S3 with AES-256. Public access blocked. DynamoDB prevents two deploys from corrupting state simultaneously.
✓ Encrypted · locked · private
💣
One-click teardown
A manual GitHub Actions workflow destroys all 41 resources across both clouds when triggered. No orphaned resources, no surprise bills after the demo ends.
✓ terraform destroy on demand
